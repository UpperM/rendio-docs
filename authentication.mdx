Rendio uses bearer-token authentication. Every request to the API must include
a valid API key in the `Authorization` header.

## Getting your API key

<Steps>
  <Step title="Open the dashboard">
    Sign in at [rendio.dev/dashboard](https://rendio.dev/dashboard).
  </Step>
  <Step title="Create a key">
    Go to **API Keys** and click **Create API Key**. Give it a descriptive name
    (e.g. `staging-backend`, `prod-billing-worker`).
  </Step>
  <Step title="Copy it now">
    The full key is shown **exactly once**. After you close the dialog you can
    only see the prefix and the last 4 characters.
  </Step>
</Steps>

<Warning>
  API keys grant full access to your account. Never expose them in client-side
  code, public repositories, or logs.
</Warning>

## Key formats

Rendio uses two key types to separate production and development:

| Format | Environment | Description |
|--------|-------------|-------------|
| `rk_live_[32 chars]` | Production | Real API calls, billed to your account |
| `rk_test_[32 chars]` | Test / Sandbox | Safe for development, no billing impact |

Both types have identical functionality. Use `rk_test_*` keys during
development to avoid consuming your production quota.

## Using your API key

Include your key in the `Authorization` header with a `Bearer` prefix:

<Tabs>
  <Tab title="curl">
    ```bash
    curl https://api.rendio.dev/v1/render \
      -H "Authorization: Bearer rk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
      -H "Content-Type: application/json" \
      -d '{"html":"<h1>Hello</h1>"}'
    ```
  </Tab>
  <Tab title="Node">
    ```ts
    import Rendio from "rendio";

    // Pass directly
    const rendio = new Rendio("rk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx");

    // Or via env (recommended)
    // export RENDIO_API_KEY=rk_live_...
    const rendio = new Rendio();
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import rendio

    # Pass directly
    client = rendio.Rendio(api_key="rk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx")

    # Or via env (recommended)
    # export RENDIO_API_KEY=rk_live_...
    client = rendio.Rendio()
    ```
  </Tab>
  <Tab title="Go">
    ```go
    // Via env (recommended) — reads RENDIO_API_KEY
    client, err := rendio.NewClient()

    // Or pass it explicitly
    client, err := rendio.NewClient(rendio.WithAPIKey("rk_live_..."))
    ```
  </Tab>
</Tabs>

### Environment variables

Store the key in an environment variable and let the SDK pick it up:

```bash
# .env — never commit this file
RENDIO_API_KEY=rk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

<Tip>
  Add `.env` to `.gitignore` to prevent accidentally committing the secret.
</Tip>

## Authentication errors

If authentication fails, the API returns **HTTP 401** with:

```json
{
  "error": {
    "code": "invalid_api_key",
    "message": "API key is required. Get one at https://rendio.dev/dashboard"
  }
}
```

Common causes:

- **Missing header** — no `Authorization` sent.
- **Bad format** — key does not match `rk_live_*` or `rk_test_*`.
- **Revoked key** — the key was deleted from the dashboard.
- **Malformed prefix** — `Bearer` missing or typo.

The SDKs throw a typed error you can branch on:

```ts
import Rendio, { AuthenticationError } from "rendio";

try {
  const rendio = new Rendio("invalid_key");
  await rendio.pdfs.create({ html: "<h1>Test</h1>" });
} catch (err) {
  if (err instanceof AuthenticationError) {
    console.error("Invalid API key:", err.message, "request:", err.requestId);
  }
}
```

## Key rotation

1. Create a new key in the dashboard.
2. Update your environment variables / secret manager.
3. Deploy the change.
4. **Verify** no traffic is using the old key (dashboard → usage).
5. Delete the old key.

<Note>
  Keys don't expire automatically — they remain valid until you delete them.
</Note>

## Security best practices

1. **Never hardcode keys** — always use environment variables or a secrets manager.
2. **Use test keys during development** — keep production quota clean.
3. **Rotate after team changes** — especially after a contributor departs.
4. **Monitor usage** — the dashboard shows per-key request counts; unusual spikes warrant attention.
5. **Minimum distribution** — only issue keys to people and systems that actually need them. One key per workload.

## Preview mode (no API key)

For quick testing, you can render PDFs without an API key. Preview mode:

- Adds a watermark to all output.
- Is rate-limited to **10 requests per hour per IP**.
- Supports `response: "buffer"` only (URL mode is disabled — there's no account to bill the storage against).

```bash
curl https://api.rendio.dev/v1/render \
  -H "Content-Type: application/json" \
  -d '{"html":"<h1>Preview</h1>","preview":true,"response":"buffer"}' \
  --output preview.pdf
```

Useful for a zero-setup try-before-you-sign-up flow. The VSCode extension uses
this under the hood for live preview.
