title: "OpenTelemetry"
description: "Zero-config distributed tracing. Rendio spans appear as children of your current trace."
---

The Rendio SDKs ship with built-in OpenTelemetry instrumentation. If you have
OTEL configured in your application, Rendio spans appear automatically in
your APM as children of your current trace. No extra setup on your end.

## How it works

OTEL is loaded as an **optional peer dependency**. If no `TracerProvider` is
registered, every instrumentation call is a zero-allocation no-op — no
overhead, no console noise. When your app registers a provider, the SDK:

1. Opens a `rendio.pdfs.create` span for each API call.
2. Opens a `rendio.pdfs.create.attempt` child span per HTTP attempt (one per
   retry).
3. Injects `traceparent` into the outbound request, so the Rendio gateway
   and worker spans link into your trace.
4. Populates `traceId` and `spanId` on every thrown `RendioError` for
   log-to-trace correlation.

## Verify it works

This self-contained Node script uses an in-memory exporter to print the span
tree. Run it to confirm the SDK emits spans before wiring a real backend.

```ts
import { NodeSDK } from "@opentelemetry/sdk-node";
import {
  InMemorySpanExporter,
  SimpleSpanProcessor,
} from "@opentelemetry/sdk-trace-base";
import Rendio from "rendio";

const exporter = new InMemorySpanExporter();
const sdk = new NodeSDK({
  spanProcessors: [new SimpleSpanProcessor(exporter)],
});
sdk.start();

const rendio = new Rendio();
try {
  await rendio.pdfs.create({ html: "<h1>Hello</h1>" });
} catch {
  // ignore — we only care about spans
}
await sdk.shutdown();

for (const span of exporter.getFinishedSpans()) {
  console.log(`[${span.name}]`, span.attributes);
}
```

Expected:

```
[rendio.pdfs.create.attempt]  http.retry_count=0  http.response.status_code=200
[rendio.pdfs.create]          rendio.mode=html    rendio.pages=1   rendio.size_bytes=14832
```

## Go setup

The Go SDK does not import `opentelemetry-go` directly. It defines a
minimal `Tracer` / `Span` interface — you inject an adapter via `WithTracer`.

```go
// The interfaces you need to implement:
type Tracer interface {
    StartSpan(ctx context.Context, name string) (Span, context.Context)
}

type Span interface {
    SetStatus(ok bool, msg string)
    SetAttribute(key string, value any)
    AddEvent(name string, attrs map[string]any)
    RecordError(err error)
    End()
    TraceParent() string
}
```

Wire it up at client construction:

```go
client, err := rendio.NewClient(
    rendio.WithTracer(myOTELAdapter),
    rendio.WithMeter(myOTELMeterAdapter), // optional — enables metrics
)
if err != nil { log.Fatal(err) }
```

The `Meter` interface:

```go
type Meter interface {
    Counter(name, description string) Counter
    Histogram(name, description, unit string) Histogram
}

type Counter interface {
    Add(value int64, attrs map[string]any)
}

type Histogram interface {
    Record(value float64, attrs map[string]any)
}
```

A full `rendio-go-otel` adapter package is planned. Until then, bridge the
interfaces above to your `go.opentelemetry.io/otel` providers. See the
[Go SDK README](https://github.com/rendio/rendio-go#opentelemetry) for a
complete adapter example.

## Attributes

### `rendio.pdfs.create` (SDK span — one per user-level call)

Attributes set by the SDK on the parent span:

| Attribute | Type | When set |
|---|---|---|
| `rendio.mode` | `"html" \| "url" \| "template"` | Always (pre-request) |
| `rendio.output` | `"pdf" \| "png" \| "jpeg"` | Always (pre-request) |
| `rendio.sdk.version` | string | Always (e.g. `rendio@1.0.0`, `rendio-go/0.1.0`, `rendio-python@0.1.0`) |
| `rendio.pages_hint` | int | When `data` keys present (template mode) |
| `rendio.pages` | int | On success (from response body) |
| `rendio.size_bytes` | int | On success (from response body) |
| `rendio.chromium_version` | string | On success (from response headers) |
| `http.response.status_code` | int | On response (success or error) |
| `error.type` | string | On error (e.g. `rate_limit`, `timeout`, `validation`, `auth`) |

All three SDKs (Node, Python, Go) emit the full attribute set above.

### `rendio.pdfs.create.attempt` (SDK span — one per HTTP attempt)

| Attribute | Type | When set |
|---|---|---|
| `http.retry_count` | int | Always (0-indexed) |
| `http.response.status_code` | int | On response |

### `rendio.pdfs.get` / `rendio.pdfs.get.attempt`

The Go and Python SDKs create spans for `pdfs.get` / `pdfs.Get` calls.
Attempt spans carry the same `http.retry_count` and
`http.response.status_code` attributes as create attempts. The Node SDK
does not currently instrument `pdfs.get`.

### Phase events (Server-Timing)

When the response includes a `Server-Timing` header, the SDK emits span
events on the parent `rendio.pdfs.create` span. Each phase becomes an event
named `rendio.phase.{name}` with a `dur_ms` attribute (float, milliseconds).
All three SDKs emit phase events.

### Metrics

All three SDKs emit three OTEL metric instruments when a `MeterProvider` is
registered:

| Instrument | Type | Unit | Attributes |
|---|---|---|---|
| `rendio.requests.total` | Counter | — | `mode` (`html`, `url`, `template`) |
| `rendio.request.duration` | Histogram | ms | `mode`, `status` (`ok`, `error`) |
| `rendio.errors.total` | Counter | — | `error.type` |

In Go, inject a `Meter` via `rendio.WithMeter(myMeter)`. The `Meter`
interface defines `Counter(name, desc)` and `Histogram(name, desc, unit)`.
When no meter is provided, metrics are a zero-allocation no-op.

### Gateway spans (server-side, visible when `traceparent` is injected)

Attributes below are set by the Rendio gateway and worker, not by your
SDK. They appear in your trace only when OTEL is configured to collect
the server-side spans linked via the injected `traceparent` header
(most OTLP exporters do this automatically).

| Attribute | Value | Notes |
|---|---|---|
| `rendio.tier` | `"free" \| "starter" \| "pro"` | Auth middleware |
| `rendio.user_id` | UUID | Auth middleware |
| `rendio.api_key_id` | UUID | Auth middleware |
| `rendio.internal` | `true` | Only emitted when `true` (low-cardinality convention) |
| `rendio.coldstart` | `true` | Only emitted when `true` |
| `rendio.pages` | int | On successful render |
| `rendio.size_bytes` | int | On successful render |
| `rendio.pool_hit` | string (`"true"` / `"false"`) | On successful render |
| `rendio.chromium_version` | string | On successful render |
| `rendio.api_key_name` | string | Auth middleware |
| `rendio.filename` | string | When `filename` param is set |

## Opt-out

Pass `telemetry: false` for a complete hard bypass — no spans, no metrics,
no `traceparent` header. Useful for batch jobs where you don't want Rendio
traffic in your trace data.

```ts
const rendio = new Rendio(process.env.RENDIO_API_KEY!, { telemetry: false });
```

## Log correlation (Pino example)

```ts
import pino from "pino";
import Rendio, { RendioError } from "rendio";

const logger = pino();
const rendio = new Rendio();

try {
  const pdf = await rendio.pdfs.create({ html: "<h1>Invoice</h1>" });
  logger.info({ pdfUrl: pdf.url }, "PDF rendered");
} catch (err) {
  if (err instanceof RendioError) {
    logger.error(
      { code: err.code, requestId: err.requestId, traceId: err.traceId, spanId: err.spanId },
      "Rendio render failed"
    );
  }
}
```

`traceId` and `spanId` are `undefined` when no active OTEL span exists
(e.g. `telemetry: false`, or OTEL not configured).

## Known issues

### Double-instrumentation with `@opentelemetry/instrumentation-undici`

If your stack auto-instruments outgoing `undici` / `fetch` calls, the HTTP
request to Rendio gets instrumented **twice** — once by undici's
auto-instrumentation, once by Rendio's per-attempt child span. Filter out
Rendio hostnames in the undici config:

```ts
import { UndiciInstrumentation } from "@opentelemetry/instrumentation-undici";

new UndiciInstrumentation({
  ignoreOutgoingRequestHook(req) {
    return req.origin?.includes("rendio.dev/api") ?? false;
  },
});
```

### Cloudflare Workers

Enable `nodejs_compat` in `wrangler.toml` — OTEL's `AsyncLocalStorage`-based
context propagation needs it.

```toml
compatibility_flags = ["nodejs_compat"]
```

Without it, `rendio.pdfs.create` still works — only the trace-tree linking
is affected.
