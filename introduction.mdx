Rendio is a managed Chromium rendering service. Send HTML (or a URL), get a
PDF, PNG, or JPEG back — either as a CDN URL or as raw bytes. No headless
browser to run, no font packages to bundle, no `--no-sandbox` in your Docker
image.

## How it works

<Steps>
  <Step title="Send a request">
    `POST /v1/render` with your HTML and render options.
  </Step>
  <Step title="Chromium renders">
    Rendio boots a warm Chromium pool, renders, and extracts bytes.
  </Step>
  <Step title="Receive a URL or buffer">
    `response: "url"` uploads the file to storage and returns a CDN link.
    `response: "buffer"` streams raw bytes back in the response body.
  </Step>
</Steps>

## What's production-grade about it

- **Typed error taxonomy.** Every failure mode has a stable `error.code`
  you can branch on — see [Error codes](/api-reference/errors).
- **Idempotency.** The `Idempotency-Key` header deduplicates retries for up
  to 24 hours.
- **OpenTelemetry.** SDKs propagate W3C `traceparent`; thrown errors expose
  `traceId` and `spanId` for log-to-trace correlation.
- **Predictable retries.** SDK clients attempt 2 retries by default, with
  exponential backoff and jitter (500&nbsp;ms base, 5&nbsp;s cap). The
  `Retry-After` header is honored when present. Non-retryable classes
  (`validation_error`, `invalid_api_key`, `quota_exceeded`, `render_failed`,
  `url_blocked`, `url_unreachable`, `url_http_error`, `url_timeout`) never
  retry.
